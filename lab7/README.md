# Руководство по использованию Recipe Manager

## Установка и запуск

### Требования
- PHP 7.4+ (для типизации свойств)
- Веб-сервер (Apache, Nginx) или встроенный PHP-сервер

### Запуск на встроенном сервере PHP
```bash
cd c:\Users\kerpi\OneDrive\Рабочий стол\php\lab7
php -S localhost:8000
```

Затем откройте в браузере: `http://localhost:8000`

## Функции приложения

### 1. Добавление рецепта
- Заполните форму на главной странице
- Все поля обязательны (title, ingredients, instructions, author)
- Название должно быть от 3 до 100 символов
- Нажмите "Сохранить"

### 2. Просмотр списка
- На главной странице отображается таблица со всеми рецептами
- Рецепты можно сортировать по названию или категории

### 3. Редактирование рецепта
- Нажмите ✏️ рядом с нужным рецептом
- Отредактируйте данные
- Нажмите "Сохранить"
- Дата создания сохраняется автоматически

### 4. Удаление рецепта
- Нажмите ❌ рядом с нужным рецептом
- Подтвердите удаление
- Рецепт будет удален из базы

## 1. Отличие нативных PHP-шаблонов от Twig

### Нативные PHP-шаблоны

Это обычные `.php` файлы, в которых HTML смешивается с PHP-кодом (`<?php ... ?>`).

**Преимущества:**

* Не требует установки дополнительных библиотек
* Полная гибкость (можно использовать весь PHP)
* Быстрое внедрение в небольших проектах

**Недостатки:**

* Легко смешать логику и представление
* Код становится трудночитаемым
* Сложнее поддерживать и масштабировать
* Повышенный риск ошибок и уязвимостей (например, забыли экранировать вывод)

---

### Twig (шаблонизатор)

Twig — это отдельный шаблонизатор со своим синтаксисом (`{{ }}`, `{% %}`).

**Преимущества:**

* Чёткое разделение логики и представления
* Чистый и читаемый код
* Автоматическая защита от XSS (экранирование)
* Поддержка фильтров, наследования, блоков
* Удобен для крупных проектов

**Недостатки:**

* Требует установки через Composer
* Нужно изучить новый синтаксис
* Ограниченная логика (нельзя использовать весь PHP напрямую)

---

## 2. Зачем разделять логику и представление?

Разделение делает проект более понятным и поддерживаемым.

**Логика (PHP):**

* Работа с данными (чтение, запись, обработка)
* Обработка запросов
* Подготовка переменных

**Представление (шаблоны):**

* Только отображение данных
* HTML + минимальная логика

---

### Проблемы при смешивании:

* Код становится "кашей"
* Сложно читать и отлаживать
* Трудно повторно использовать части кода
* Дизайнер не сможет работать без знания PHP
* Больше багов и уязвимостей

👉 Итог: разделение делает код чище, безопаснее и удобнее для развития.

## 3. Наследование шаблонов в Twig

Наследование позволяет создать базовый шаблон и переиспользовать его.

---

### {% extends %}

Указывает родительский шаблон:

```twig id="x2k9va"
{% extends "base.html.twig" %}
```

👉 Текущий шаблон будет использовать структуру базового.

---

### {% block %}

Блоки — это части шаблона, которые можно переопределять.

**Базовый шаблон:**

```twig id="zjz5ch"
<!DOCTYPE html>
<html>
<head>
    <title>{% block title %}Сайт{% endblock %}</title>
</head>
<body>
    {% block content %}{% endblock %}
</body>
</html>
```

**Дочерний шаблон:**

```twig id="3rzg0m"
{% extends "base.html.twig" %}

{% block title %}Список рецептов{% endblock %}

{% block content %}
    <h1>Рецепты</h1>
{% endblock %}
```

---

### Как это работает:

* Базовый шаблон задаёт структуру
* Дочерний подставляет содержимое в блоки
* Twig объединяет их в итоговый HTML

---

### Преимущества наследования:

* Нет дублирования кода
* Удобно менять общий дизайн
* Чистая структура шаблонов
* Легко масштабировать проект

---

## Вывод

* Нативные шаблоны — проще, но менее структурированы
* Twig — мощнее и удобнее для больших проектов
* Разделение логики и представления — обязательная практика
* Наследование в Twig делает код гибким и переиспользуемым
